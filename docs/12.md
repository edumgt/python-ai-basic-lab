## 프로젝트할 때 자주 쓰는 말 풀이

| 단어 | 쉬운 뜻 | 프로젝트에서 하는 일 |
|------|------|------|
| 가설 | "아마 이럴 것 같다"는 예상 | 예: 거래량 특성이 중요할 것 같다고 먼저 생각해봅니다. |
| 비교 | 둘 이상을 나란히 보는 것 | 모델, 회사, 지표를 서로 놓고 판단합니다. |
| 근거 | 그렇게 말한 이유 | `AUC`, 중요도, 차트가 근거가 됩니다. |
| 해석 | 숫자를 사람 말로 바꾸기 | "거래량 급증을 모델이 중요하게 봤다"처럼 설명합니다. |
| 결론 | 마지막에 내리는 판단 | 어떤 모델을 먼저 써볼지 정리합니다. |
| 후속 실험 | 다음에 더 해볼 일 | 특성 추가, 기간 변경, 다른 모델 비교 등이 됩니다. |

## 최종 정리 템플릿

아래 표를 채우면 작은 프로젝트 보고서가 됩니다.

| 항목 | 내가 적을 내용 |
|------|------|
| 분석한 데이터/샘플 |  |
| 비교한 모델 |  |
| 가장 좋았던 모델 |  |
| 근거가 된 지표 |  |
| 중요 특성 1위 |  |
| 해석 한 줄 |  |
| 남은 의문 |  |

### 보고서를 쓸 때 한 줄씩 어떻게 적으면 좋을까요?

- `가장 좋았던 모델`: 점수가 가장 높았던 모델 이름만 쓰지 말고, 왜 좋았는지도 같이 적습니다.
- `근거가 된 지표`: accuracy 하나만 쓰기보다 AUC, precision, 중요도까지 함께 적으면 더 탄탄합니다.
- `해석 한 줄`: "거래량 변화가 큰 날을 모델이 중요하게 봤다"처럼 사람 말로 바꿔 적습니다.
- `남은 의문`: "기간을 바꾸면 결과가 같을까?" 같은 다음 질문을 적으면 좋은 프로젝트가 됩니다.

## 이 저장소에서 문서와 웹앱을 쓰는 가장 좋은 방법

1. 개념 문서 1개를 읽습니다.
2. 바로 연결된 웹앱을 엽니다.
3. 숫자를 기록합니다.
4. 모델을 하나 더 바꿔 비교합니다.
5. 마지막에 한 줄 해석을 남깁니다.

이렇게 해야 문서가 실제 실습으로 이어집니다.

## 막히면 이렇게 생각하면 됩니다

1. 먼저 문제를 말로 바꿉니다.
   예: "내일 오를까?"는 분류 문제입니다.
2. 그다음 데이터를 확인합니다.
   예: 가격, 거래량, 회사명이 있는지 봅니다.
3. 모델을 하나 실행합니다.
4. 숫자와 차트를 보고 이유를 적습니다.
5. 다른 모델과 비교해 더 나은지 확인합니다.

이 순서만 지켜도 작은 퀀트 프로젝트를 충분히 해낼 수 있습니다.
